## 1. 背景・前提

Phase 11.5-F の設計で「マネージャースタッフ」の型定義を行った。
ここでは「マネージャーをどう管理・育成するか」の3案を詳細設計する。

現状のマネージャーの役割:
- 対戦相手選手のスカウティング評価（言語化・精度可変）
- ランク: F〜S の7段階
- スペシャリティ: scouting / mental_care / data_analysis / general

---

## 2. 案A — マネージャー育成

### 2.1 概念

1人のマネージャーを長期間育成する。
試合経験・練習参加・研修などで経験値が貯まり、ランクアップする。

### 2.2 型定義

```ts
interface ManagerStaff {
  id: string;
  firstName: string;
  lastName: string;
  rank: ManagerRank;          // F〜S
  experience: number;          // 0〜100（100でランクアップ）
  specialty: ManagerSpecialty;
  grade: 1 | 2 | 3;           // 学年（3年で卒業→後任登場）
  traits: ManagerTraitId[];   // 特性（偵察向き、ムードメーカー等）
}

type ManagerTraitId =
  | 'scout_eye'         // 偵察の目: 評価精度+10%
  | 'mood_maker'        // ムードメーカー: 選手モチベーション全体+3
  | 'data_nerd'         // データオタク: 試合後の評価精度+15%
  | 'energetic'         // 元気印: 毎日の経験値取得+20%
  | 'shy'               // 引っ込み思案: 経験値取得-10%
  | 'veteran_student';  // OBの孫: 初期ランクがC以上で固定
```

### 2.3 経験値取得ロジック

```
日次処理で以下の経験値を加算:
  通常練習日:   +1
  試合日（観戦）: +3
  大会試合（観戦）: +5
  視察行動：    +8

ランクアップ閾値:
  F→E: 100exp
  E→D: 150exp
  D→C: 200exp
  C→B: 300exp
  B→A: 400exp
  A→S: 500exp

ランクアップ時:
  新しいスペシャリティが解放されることがある（ランダム）
```

### 2.4 UI

```
/play/team 画面内「スタッフ」タブ or /play/staff ページ

┌──────────────────────────────────────┐
│  マネージャー: 佐藤 花              │
│  学年: 2年  ランク: C               │
│  特性: データオタク、ムードメーカー  │
│  経験値: ████████░░ 78/100          │
│  次のランクまであと 22 exp           │
│                                     │
│  [ランクアップ条件を確認]            │
└──────────────────────────────────────┘
```

### 2.5 メリット・デメリット

| 項目 | 内容 |
|---|---|
| ✅ メリット | 1人のキャラクターへの愛着が生まれる。成長の実感がある。シンプルで迷わない |
| ✅ メリット | 卒業→後任登場のドラマが生まれる |
| ❌ デメリット | 役割が1つに限定（スカウト専任 or メンタルケア専任の選択ができない）|
| ❌ デメリット | スカウト機能との棲み分けが若干曖昧になる |
| 📦 実装コスト | **小**（1〜2日）。既存マネージャー型の拡張のみ |

## 3. 案B — マネージャー複数雇用

### 3.1 概念

複数のマネージャーを「役割分担」で雇用する。
プレイヤーは「誰に何を担当させるか」を決める戦略的要素が生まれる。

### 3.2 役割定義

```ts
type ManagerRole =
  | 'head'        // チーフマネージャー（1人、全体統括）
  | 'scout'       // スカウト担当（対戦相手評価・視察）
  | 'mental'      // 選手メンタルケア担当（モチベーション管理）
  | 'analyst';    // データ分析担当（成長・コンディション分析）
```

### 3.3 雇用・管理

```ts
interface ManagerStaffList {
  /** 在籍中のマネージャーリスト */
  members: ManagerStaff[];
  /** 役割アサイン */
  assignments: Record<ManagerRole, string | null>;  // role → managerId
  /** 上限人数（学校のランクに応じて変化） */
  maxMembers: number;  // 初期 2名、評判Aで3名、評判Sで4名
}
```

### 3.4 役割効果

| 役割 | 担当マネージャーがいない | いる（Cランク） | いる（Sランク） |
|---|---|---|---|
| スカウト | 評価1項目・誤差40% | 3〜4項目・誤差15% | 6〜7項目・誤差2% |
| メンタルケア | 効果なし | 全選手やる気+2/日 | 全選手やる気+5/日 |
| データ分析 | 成長グラフなし | 直近30日の成長可視化 | 詳細成長分析 |

### 3.5 UI

```
/play/staff 新設ページ

┌──────────────────────────────────────────┐
│  スタッフ管理                             │
│  在籍 2名 / 上限 2名                      │
├──────────────────────────────────────────┤
│  [チーフ] 佐藤 花 (C)                    │
│  [スカウト] 田中 葵 (B) ← スカウト担当   │
│  [メンタル] (未配置)                      │
│  [分析] (未配置)                          │
├──────────────────────────────────────────┤
│  [新しいマネージャーを探す] ← 次年度勧誘  │
└──────────────────────────────────────────┘
```

### 3.6 メリット・デメリット

| 項目 | 内容 |
|---|---|
| ✅ メリット | 戦略的な役割分担の楽しさ。「スカウト重視か、メンタルケア重視か」の選択 |
| ✅ メリット | 複数キャラへの愛着が生まれる。マネージャーのドラマが豊かになる |
| ❌ デメリット | 管理が複雑になる。初心者には敷居が高い |
| ❌ デメリット | 実装コストが高い（複数人管理・役割アサイン UI 等） |
| 📦 実装コスト | **大**（3〜5日）。専用管理画面・役割アサインロジック等 |

## 4. 案C — 統合管理画面（/play/staff）

### 4.1 概念

コーチ・マネージャー・監督を一覧で管理する「スタッフ管理画面」を新設する。
現状の「監督」データも統合して表示し、将来的にコーチ雇用機能も見据えた画面設計。

### 4.2 スタッフタイプ

```ts
type StaffType = 'manager' | 'coach' | 'head_coach';

interface StaffMember {
  id: string;
  type: StaffType;
  name: string;
  rank: ManagerRank | CoachRank;
  specialty: string;
  // コーチの場合: 担当ポジション（打撃コーチ、投手コーチ等）
  coachSpecialty?: CoachSpecialty;
}

type CoachSpecialty = 'batting' | 'pitching' | 'defense' | 'physical' | 'mental';
```

### 4.3 UI

```
/play/staff ページ

┌──────────────────────────────────────────┐
│  スタッフ一覧                             │
```

### 4.4 メリット・デメリット

| 項目 | 内容 |
|---|---|
| ✅ メリット | 将来のコーチ雇用機能への拡張性が高い。一箇所でスタッフを管理できる |
| ✅ メリット | 現状の監督データを活用できる |
| ❌ デメリット | 現時点ではマネージャーだけなので「空っぽ感」がある |
| ❌ デメリット | コーチ機能が未実装なら中途半端な画面になる |
| 📦 実装コスト | **中**（2〜3日）。画面新設 + 監督データとの統合 |

## 6. 推奨案

### **推奨: 案Aをベースに、案Cの画面設計を組み込む（A + C ハイブリッド）**

**理由:**

1. **今すぐ楽しめる**: 案Aの育成要素は実装コストが低く、Phase 11.5 内で完結できる

2. **将来への布石**: 案Cの `/play/staff` 画面を作っておくことで、後から案Bの複数雇用や、コーチ雇用（Phase 12 候補）を自然に追加できる

3. **段階的導入**: まず1人のマネージャーを育てる体験から始め、需要があれば複数雇用に拡張できる

### ハイブリッド案の実装

```
Phase 11.5-G MVP（2〜3日）:
  - /play/staff 画面を新設
  - 画面に「マネージャー」1名の育成 UI を表示
  - 経験値・ランクアップ・特性の実装

将来（Phase 12候補）:
  - コーチ雇用機能を追加（/play/staff の「コーチ」枠）
  - マネージャー複数雇用（案Bの要素）を追加
```
