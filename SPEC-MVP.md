## 1. ゲームコンセプト定義

### 1.1 ワンライナー

> プレイヤーは高校野球部の監督として、選手を育て、采配を振り、甲子園を目指す——何十年でも続けられる長期シミュレーション。

### 1.2 ジャンル

- 育成×采配×シミュレーション
- ターン進行型（日単位 / イベント単位）
- ストーリー自動生成型（選手の人生がドラマになる）

### 1.3 設計理念

| 原則 | 説明 |
|------|------|
| **監督視点** | プレイヤーは監督。選手を直接操作しない。指示を出し、選手が応える（応えないこともある） |
| **育成が中心** | 学校経営は主目的にしない。中心は育成・采配・試合・3年間のドラマ |
| **時間は流れる** | 通常は時短進行。気になる選手・局面だけ深掘り可能 |
| **永続性** | 何十年でもプレイ可能。卒業生は消えず、OBとして世界に残る |
| **不確実性** | 能力だけで勝てない。性格・精神・コンディション・運が結果を左右する |

### 1.4 リファレンス

- 栄冠ナイン（パワプロ）: 監督視点・年間進行の基本構造
- Football Manager: 長期運営・選手成長・スカウトの深さ
- ウイニングポスト: 世代交代・OB追跡・長期シミュレーション
- 俺の甲子園: 高校野球シムとしてのベースライン

## 2. プレイヤー体験の定義

### 2.1 監督の役割

プレイヤーが操作・判断するもの：

| カテゴリ | 操作内容 |
|----------|----------|
| **育成方針** | 練習メニュー策定、個別指導方針、合宿計画 |
| **采配** | 打順・守備位置決定、試合中の継投/代打/バント等の指示 |
| **スカウト** | 中学大会の視察指示、注目選手の追跡、勧誘 |
| **マネジメント** | ベンチ入りメンバー選定、キャプテン指名、問題対応 |
| **深掘り** | 任意の選手の詳細画面を開いて状態・履歴・関係を確認 |

### 2.2 プレイヤーが操作しないもの

- 選手の個々の動き（試合中の打撃・投球・守備はAI自動）
- 学校の運営・予算・施設（MVPでは固定）
- 対戦相手の操作

### 2.3 体験の温度感

```
通常進行: サクサク → 1日が数秒〜十数秒で流れる
イベント発生: ポップアップ → 判断 → 結果
試合: 最もリッチ → 2D表現で打席・投球を可視化
深掘り: プレイヤー任意 → 選手の詳細を見る、過去を辿る
```

## 3. コアループ

### 3.1 デイリーループ（最小単位）

```
┌─────────────────────────────────────┐
│  1日の開始                           │
│  ├─ 朝: コンディション判定           │
│  ├─ 日中: 授業（スキップ/イベント）   │
│  ├─ 放課後: 練習 → 能力変動          │
│  ├─ イベント発生判定                  │
│  │   ├─ 個人イベント（怪我/恋愛/喧嘩等）│
│  │   ├─ チームイベント（士気/雰囲気）  │
│  │   └─ 外部イベント（スカウト/OB来訪） │
│  └─ 日終了 → 次の日へ               │
└─────────────────────────────────────┘
```

### 3.2 試合ループ（詳細は §8）

```
┌─────────────────────────────────────┐
│  試合開始                            │
│  ├─ イニングループ                   │
│  │   ├─ 攻撃: 打席ごとに采配判断     │
│  │   ├─ 守備: 投球方針・守備配置     │
│  │   └─ 結果: 2D演出                │
│  ├─ 采配介入ポイント                 │
│  │   ├─ 代打/代走/継投              │
│  │   ├─ バント/盗塁/エンドラン指示   │
│  │   └─ マウンド訪問（精神回復）     │
│  └─ 試合終了 → 成長判定（最大）     │
└─────────────────────────────────────┘
```

### 3.3 年間ループ（詳細は §4）

```
4月 → ... → 3月 → 年度替わり（卒業/入学）→ 次の年度へ
```

### 3.4 世代ループ（詳細は §5）

```
N年目 → N+1年目 → ... → 永続
├─ 卒業生 → OBデータベースへ
├─ 新入生 → スカウト結果 + 自動入部
└─ 監督の評判・実績が蓄積
```

## 4. 1年の流れ

### 4.1 年間カレンダー

| 月 | 主要イベント | ゲーム上の意味 |
|----|-------------|---------------|
| **4月** | 入学式、新入部員入部、チーム編成 | 新戦力合流。1年生の適性見極め期間 |
| **5月** | 春季大会（地区） | 公式戦①。夏への布石。新チーム初陣 |
| **6月** | 練習試合期間、梅雨 | 育成集中期。怪我リスク↑、士気管理 |
| **7月** | 夏の地方大会 | **年間最大イベント**。一発勝負トーナメント |
| **8月前半** | 甲子園（勝ち上がった場合） | 全国大会。最高の成長機会 |
| **8月後半** | 3年生引退、新チーム発足、夏合宿 | 世代交代。合宿で集中育成 |
| **9月** | 秋季大会（地区） | センバツ選考に直結 |
| **10月** | 秋季大会（地方）、ドラフト会議 | 上位進出→センバツ有力。3年生ドラフト |
| **11月** | 明治神宮大会（該当チーム）、オフ期間開始 | 秋の頂点。オフで体力回復・基礎鍛錬 |
| **12月** | オフ期間、自主練、中学生スカウト本格化 | スカウト活動の重点期間 |
| **1月** | センバツ出場校発表、冬季練習 | 吉報 or 落選。精神面のイベント |
| **2月** | 春季キャンプ、紅白戦 | 実戦形式で仕上げ |
| **3月** | センバツ甲子園、卒業式 | 春の甲子園 + 卒業ドラマ |

### 4.2 大会構造

```
春季大会（5月）
├─ 地区大会 → 都道府県大会
└─ 勝敗は夏のシード権に影響

夏の大会（7月〜8月）
├─ 地方大会（都道府県トーナメント）
└─ 甲子園（全国大会） ← ゲームのクライマックス

秋季大会（9月〜11月）
├─ 地区大会 → 地方大会
├─ 明治神宮大会（地方大会優勝校）
└─ 成績がセンバツ選考に直結

センバツ甲子園（3月）
├─ 秋の成績ベースで選出
└─ 招待制（選ばれるかどうか自体がイベント）
```

### 4.3 特殊イベント

| イベント | 時期 | 内容 |
|----------|------|------|
| 合宿 | 8月、12月〜1月 | 集中育成。能力上昇率が通常練習の1.5〜2倍 |
| 練習試合 | 通年（大会期間外） | 対戦相手を選べる。試合経験を積める |
| 高校日本代表選出 | 不定期 | 有力選手が選出される。経験値大+評判上昇 |
| OB来訪 | ランダム | 卒業生が訪問。チーム士気UP。特殊指導イベント |
| スカウト陣来校 | 秋〜冬 | プロスカウトが試合を見に来る。選手の緊張度に影響 |

### 4.4 進行速度

| フェーズ | 進行速度 | 備考 |
|----------|---------|------|
| 通常日 | 自動早送り（1日＝数秒） | イベント発生時のみ停止 |
| 練習日 | 早送り可 | 練習メニュー結果をサマリ表示 |
| 大会期間 | 試合ごとに詳細表示 | 自チームの試合は2D演出 |
| 試合中 | リアルタイム進行 | 打席単位で采配介入可能 |

## 5. 3年間の流れ

### 5.1 学年ごとの体験設計

#### 1年生（見習い期）

| 要素 | 内容 |
|------|------|
| 立場 | ベンチ外が基本。練習参加のみ |
| 育成 | 基礎能力の伸びが最大。ポテンシャルが徐々に可視化される |
| イベント | 先輩との関係構築、寮生活への適応、ホームシック、初試合の緊張 |
| 監督操作 | 1年生の適性見極め。秋以降に有望株をベンチ入りさせるかの判断 |
| 物語的意味 | 「原石の発見」。3年後のエースや四番がここにいるかもしれない |

#### 2年生（成長期）

| 要素 | 内容 |
|------|------|
| 立場 | レギュラー候補〜主力。チームの中核になり始める |
| 育成 | 専門能力の伸びが最大。変化球習得、打撃フォーム固めなど |
| イベント | スランプ、ポジション争い、後輩指導、恋愛トラブル、チーム内対立 |
| 監督操作 | 来年のチーム構想。キャプテン候補の見極め |
| 物語的意味 | 「葛藤と成長」。挫折と突破が最も多い学年 |

#### 3年生（集大成）

| 要素 | 内容 |
|------|------|
| 立場 | チームリーダー。最後の夏に向けた全力 |
| 育成 | 能力の伸びは鈍化。ただし精神面・試合勘は最大 |
| イベント | 最後の夏への覚悟、引退、卒業、進路決定、ドラフト |
| 監督操作 | 最終打順・戦術確定。進路相談対応 |
| 物語的意味 | 「有終の美 or 悔し涙」。プレイヤーが最も感情移入する局面 |

### 5.2 卒業・進路システム

```
3年生（秋〜冬）
├─ ドラフト指名 → プロ入り
├─ 大学進学（野球推薦 / 一般）
├─ 社会人野球
├─ 就職（野球引退）
└─ 浪人・その他
```

#### 進路決定の要素

| 要素 | 影響 |
|------|------|
| 能力値 | 高ければプロ・強豪大学からオファー |
| 実績 | 甲子園出場、大会成績が進路に直結 |
| 性格 | 野球継続意欲、上昇志向が影響 |
| 監督の推薦 | プレイヤーが推薦状を書くイベント |
| 怪我歴 | 慢性的な怪我があるとプロ評価↓ |

### 5.3 卒業後追跡（OBシステム）

```
OBデータベース
├─ プロ入り → 年ごとの成績を自動生成
│   ├─ 1軍昇格、タイトル、引退など
│   └─ ニュースとしてゲーム内に通知
├─ 大学野球 → 大会成績を自動生成
├─ 社会人野球 → 都市対抗などに出場
└─ 引退後
    ├─ 指導者（母校に戻ってくる可能性）
    ├─ 解説者
    └─ 一般社会人
```

#### OBイベント例

| イベント | 条件 | 効果 |
|----------|------|------|
| OBがプロ初勝利 | OBがプロ投手 | チーム士気UP、学校の評判UP |
| OBが母校訪問 | ランダム+好感度 | 特別練習イベント、選手のモチベUP |
| OBからの寄付 | OBがプロで活躍中 | 設備改善（将来的な拡張要素） |
| OBの引退ニュース | OBの年齢・成績条件 | 選手への教訓イベント |

## 6. 選手個性システム

### 6.1 選手データ構造

```
Player {
  // === 基本情報 ===
  id: UUID
  name: string              // 姓名
  generation: number         // 入学年度
  grade: 1 | 2 | 3          // 学年
  position: Position         // 守備位置（複数可）
  batting: "left" | "right" | "switch"
  throwing: "left" | "right"

  // === 身体 ===
  height: number             // cm
  weight: number             // kg
  physicalType: PhysicalType // 体型分類
  appearance: AppearanceData // 容姿データ（後述）

  // === 能力値 ===
  stats: PlayerStats         // 野球能力（後述）
  potential: PotentialStats   // 潜在能力（監督には一部隠蔽）
  condition: ConditionState   // 当日のコンディション

  // === 内面 ===
  personality: Personality    // 性格特性（後述）
  mentality: MentalState     // 精神状態
  motivation: number         // モチベーション (0-100)
  
  // === 背景 ===
  background: Background     // 出身・家庭環境
  relationships: Relationship[] // 人間関係
  history: EventLog[]        // イベント履歴
  
  // === 進路 ===
  careerPath: CareerPath | null  // 卒業後の進路
}
```

### 6.2 能力値体系

#### 共通能力

| 能力 | 範囲 | 説明 |
|------|------|------|
| 体力 | 1-100 | スタミナ。練習量・試合持久力に影響 |
| 走力 | 1-100 | 足の速さ。盗塁・守備範囲・内野安打に影響 |
| 肩力 | 1-100 | 送球力。守備・盗塁阻止に影響 |
| 守備力 | 1-100 | 守備の上手さ。エラー率・守備範囲に影響 |
| 集中力 | 1-100 | 大事な場面での発揮率 |
| メンタル | 1-100 | プレッシャー耐性 |

#### 打撃能力

| 能力 | 範囲 | 説明 |
|------|------|------|
| ミート | 1-100 | バットにボールを当てる技術 |
| パワー | 1-100 | 打球の飛距離・速度 |
| 選球眼 | 1-100 | ボール球を見極める力 |
| 打撃技術 | 1-100 | 流し打ち、引っ張りなどの技術 |

#### 投球能力（投手のみ）

| 能力 | 範囲 | 説明 |
|------|------|------|
| 球速 | 80-160 | km/h単位のストレート最速 |
| コントロール | 1-100 | 投球の精度 |
| スタミナ | 1-100 | 投球持久力（打者の体力とは別枠） |
| 変化球 | Map<BallType, 1-7> | 球種ごとのキレ（1=覚えたて、7=超一流） |

#### 変化球種一覧（MVP）

| 種類 | 分類 |
|------|------|
| カーブ | 縦変化 |
| スライダー | 横変化 |
| フォーク | 落ちる球 |
| チェンジアップ | 緩急 |
| カットボール | 小さい変化 |
| シンカー/スクリュー | 利き腕方向 |
| ナックル | 特殊（MVP後） |

### 6.3 潜在能力（ポテンシャル）

```
PotentialStats {
  ceiling: PlayerStats    // 到達可能な最大値
  growthRate: number      // 成長速度係数 (0.5〜2.0)
  growthType: GrowthType  // 早熟 | 普通 | 晩成 | 天才
  hiddenTraits: Trait[]   // 隠し特性（条件で発現）
}
```

- 監督からはポテンシャルは「おおまかな印象」でしか見えない
- スカウト評価、練習での観察、試合での発揮で徐々に精度が上がる
- 「思ったより伸びなかった」「予想以上に化けた」が起こる設計

### 6.4 性格システム

#### 性格特性（Personality Traits）

各選手は以下から複数の特性を持つ：

| カテゴリ | 特性例 | 影響 |
|----------|--------|------|
| **気質** | 熱血、冷静、マイペース、繊細、豪快 | 試合中のプレッシャー反応 |
| **社交性** | リーダー気質、ムードメーカー、一匹狼、人見知り | チーム内関係、キャプテン適性 |
| **野球観** | 努力家、天才肌、戦略家、負けず嫌い、楽しむ派 | 練習効率、試合姿勢 |
| **問題傾向** | 短気、サボり癖、自信過剰、自己否定、反抗的 | 問題イベント発生率 |
| **美徳** | 責任感、思いやり、根性、素直、向上心 | 成長補正、チーム貢献 |

#### 性格の影響例

```
サイン無視率 = f(反抗的, 自信過剰, 監督との信頼度, 場面の緊張度)
練習効率 = f(努力家, モチベーション, 体調, チーム雰囲気)
大舞台での実力発揮 = f(メンタル, 集中力, 性格特性, 経験値)
```

### 6.5 精神状態（MentalState）

```
MentalState {
  mood: "絶好調" | "好調" | "普通" | "不調" | "絶不調"
  stress: number          // 0-100（高いと問題イベント↑）
  confidence: number      // 0-100（成功で↑、失敗で↓）
  teamChemistry: number   // チーム内での居心地 (0-100)
  
  // 一時的状態
  activeFlags: [
    "スランプ",
    "ゾーン",
    "怪我不安",
    "恋愛中",
    "家庭問題",
    "チーム内対立"
  ]
}
```

### 6.6 背景・バックストーリー

```
Background {
  hometown: string         // 出身地
  middleSchool: string     // 出身中学
  familyType: FamilyType   // 家庭環境分類
  economicLevel: 1-5       // 経済状況
  
  backstory: string        // 自動生成される背景テキスト
  // 例: 「父親が元高校球児。幼少期から野球漬けの生活。
  //       中学では県大会ベスト4のエースだった。
  //       家庭の経済事情で私立は選べず、公立校を選んだ。」
  
  entryReason: string      // この高校を選んだ理由
  dream: string            // 将来の夢
}
```

### 6.7 人間関係

```
Relationship {

  type: "友情" | "ライバル" | "恋愛" | "対立" | "尊敬" | "嫌悪"
  strength: number         // 関係の強さ (0-100)
  history: string[]        // 関係の経緯
```

### 6.9 イベントシステム

#### イベントカテゴリ

| カテゴリ | 例 | 発生条件 |
|----------|-----|----------|
| **日常** | 自主練報告、食堂でのエピソード | 毎日低確率 |

## 7. 中学生スカウトシステム

### 7.1 概要

毎年の新入生は以下の2ルートで獲得：

1. **自動入部**: 地域の中学生が勝手に入部してくる（学校の評判依存）
2. **スカウト**: 監督が能動的に中学生を発見・追跡・勧誘する

### 7.2 スカウト活動の流れ

```
中学大会シーズン（夏〜秋）
├─ ① 視察指示: コーチ/副部長を大会に派遣
│   ├─ 地区を選択（近隣 / 他県）
│   └─ 派遣人数で精度が変わる
├─ ② レポート受領: 注目選手のスカウトレポート
│   ├─ 能力の「印象」（正確ではない）
│   ├─ 性格の「印象」
│   └─ 他校の注目度
├─ ③ 追跡: 気になる選手を「追跡リスト」に追加
│   ├─ 複数回視察で情報精度UP
│   └─ 追跡人数には上限あり
├─ ④ 勧誘: 入学を打診
│   ├─ 学校の評判、実績、監督の知名度が影響
│   ├─ 他校との競合あり
│   └─ 家庭の事情で断られることもある
└─ ⑤ 結果: 4月に入学 or 他校へ
```

### 7.3 スカウトレポート

```
ScoutReport {
  playerId: UUID
  scoutedBy: "head_coach" | "assistant" | "coach"
  reliability: number        // レポート信頼度 (0-100)
  
  // 評価（印象ベース。実際の値とは誤差あり）
  estimatedAbility: {
    overall: "S" | "A" | "B" | "C" | "D"
    batting: "S" | "A" | "B" | "C" | "D"
    pitching: "S" | "A" | "B" | "C" | "D" | null
    defense: "S" | "A" | "B" | "C" | "D"

  estimatedPotential: "超大型" | "大型" | "普通" | "小型"
  personalityNote: string    // 「真面目で努力家」「才能はあるが気分屋」等
  
  comment: string            // 自由記述
  // 例: 「左腕から130km台のストレートとキレのあるスライダー。
  //       制球に課題あり。精神的にはまだ幼い印象。
  //       地元の強豪校も注目しているとの情報あり。」
  
  competingSchools: string[] // 競合校
}
```

### 7.4 学校の評判

スカウト成功率に大きく影響する要素：

- 能力分布: 学校の評判が高いほど優秀な選手が多い

## 8. 試合システム概要

### 8.1 表現方式

- **2Dトップビュー**（フィールド全体）+ **投球時ズームイン**（ストライクゾーン）
- リアルタイム演出ではなく、結果確定後にアニメーション再生

### 8.2 試合の進行

```
試合進行フロー:
1. スタメン・打順決定（試合前）
2. イニング開始
3. 打席ごとの処理:
   a. 投球フェーズ
      - 投手の球種・コース選択（AI自動）
      - 打者のスイング判定（AI自動）
      - 結果確定: ストライク/ボール/ファウル/インプレー
   b. 采配介入ポイント（該当時）
      - バント指示、盗塁指示、エンドラン
      - 代打、代走
      - 投手交代、マウンド訪問
   c. 2D演出
      - 投球コース表示（ストライクゾーン上）
      - 打球方向・速度の表示（フィールド上）
      - 守備の動き表示
4. イニング終了
5. 9回終了 or 延長 → 試合終了
```

### 8.3 投球の可視化

```
ストライクゾーン表示:
┌─────────────────┐
│  ①  │  ②  │  ③  │  ← 高め
├─────┼─────┼─────┤
│  ④  │  ⑤  │  ⑥  │  ← 中段
├─────┼─────┼─────┤
│  ⑦  │  ⑧  │  ⑨  │  ← 低め
└─────────────────┘
  内角    真中    外角

+ ゾーン外（ボールゾーン）も4方向で表示
+ 球筋の軌道を曲線で表示（変化球の変化量を可視化）
+ 打者のスイング軌道と接触ポイントを表示
```

### 8.4 打球の可視化

```
フィールド表示（2Dトップビュー）:
        CF
    LF      RF
  SS    2B
3B          1B
        P
        C
    [打者]

- 打球の方向と速度を矢印/ラインで表示
- 守備選手の移動を表示
- ヒット/アウトの判定結果を表示
```

### 8.5 采配システム

#### 試合前

| 采配 | 内容 |
|------|------|
| スタメン決定 | 9人+DH（大会ルール依存）の打順・守備位置 |
| ベンチメンバー | 控え選手の選定 |
| 先発投手 | 投手の選択 |
| 作戦方針 | 「打撃重視」「守備重視」「バント多用」等の全体方針 |

#### 試合中

| 采配 | タイミング | 影響 |
|------|-----------|------|
| バント指示 | 打席前 | 選手のバント技術+性格で成功率変動 |
| 盗塁指示 | ランナー発生時 | 走力+相手捕手の肩+投手のクイック |
| エンドラン | ランナー発生時 | ハイリスク・ハイリターン |
| 代打 | 打席前 | ベンチの選手を投入 |
| 代走 | 出塁時 | 足の速い選手に交代 |
| 投手交代 | 投球間 | 疲労・被打率を見て判断 |
| 守備交代 | イニング間 | 守備固め |
| マウンド訪問 | 投球間 | 投手の精神回復。回数制限あり |
| 敬遠 | 打席前 | 強打者を避ける |
| 守備シフト | イニング間 | 前進守備、バントシフト等 |

### 8.6 采配とサイン無視

監督の指示は「絶対」ではない：

```
サイン実行率 = 基本実行率 × 信頼度補正 × 性格補正 × 場面補正

基本実行率: 90%
信頼度補正: 監督との関係が良い +5%、悪い -15%
性格補正: 素直 +5%、反抗的 -10%、自信過剰（好調時）-5%
場面補正: チャンスで盗塁サイン → 慎重な選手は無視しやすい
```

- サイン無視は「悪いこと」だけではない
  - 自信過剰な四番が「バント無視→ホームラン」もありうる
  - 物語的に面白いイベントになる

### 8.7 試合結果と成長

試合は**最も成長率が高い**活動：

| 要素 | 成長量 |
|------|--------|
| 打席に立つ | 打撃経験値 ×2.0（練習比） |
| 投球する | 投球経験値 ×2.0 |
| 重要な場面 | 経験値にボーナス（得点圏、クロスゲーム等） |
| 勝利 | メンタル +、チーム士気 + |
| 敗北 | メンタル ±（性格依存）、悔しさが成長の糧に |
| 甲子園 | すべての経験値 ×3.0 |

| 判断 | 選択 | 理由 |
|------|------|------|
| サーバーサイド | **MVP: なし → v1.5: 認証APIのみ → v2: フル** | 段階的に構築 |
| クライアントDB | **IndexedDB (MVP) → SQLite (アプリ版)** | アダプターパターンで切り替え |
| AI/LLM | **不使用（MVP）** | イベントテキストはテンプレート+パラメータで生成 |
| マルチプラットフォーム | **Web先行 → Capacitorでアプリ化** | エンジン・UIを共有。ネイティブコード最小 |
| 認証タイミング | **起動時 + 定期チェック（24時間ごと）** | ゲーム体験を阻害しない |
| オフライン猶予 | **認証後72時間（3日間）はオフラインプレイ可** | 通勤・旅行中のプレイを保証 |
| セーブデータ所有権 | **ローカルが正。クラウドはバックアップ** | ユーザー体験優先。ローカル優先マージ |

### 12.6 認証・ライセンス・オフラインプレイ仕様

#### ライセンスモデル

| 項目 | 仕様 |
|------|------|
| **課金形態** | サブスクリプション（月額/年額）を想定。買い切りも検討可 |
| **認証方式** | メールアドレス + パスワード（OAuth将来対応） |
| **トークン** | JWT (アクセストークン: 1時間 + リフレッシュトークン: 30日) |
| **ライセンストークン** | 課金状態を含むJWT。有効期限=オフライン猶予期限 |
| **オフライン猶予** | 最終認証成功から **72時間（3日間）** |
| **認証チェックタイミング** | アプリ起動時 / 24時間経過ごと / 手動更新 |

#### オフラインプレイの挙動

```
アプリ起動
  │
  ├─ オンライン接続あり
  │   ├─ 認証成功 + 課金有効 → プレイ可能（猶予タイマーリセット）
  │   ├─ 認証成功 + 課金失効 → ★プレイ不可（課金案内画面）
  │   └─ 認証失敗（サーバーエラー）→ キャッシュトークンで猶予判定
  │
  └─ オフライン
      ├─ キャッシュトークンあり + 猶予期間内 → プレイ可能
      ├─ キャッシュトークンあり + 猶予期間切れ → ★プレイ不可（接続要求画面）
      └─ キャッシュトークンなし → ★プレイ不可（ログイン要求画面）

★プレイ不可 = ゲーム進行が停止。セーブデータの閲覧は可能（ロックイン防止）
```

#### ライセンストークンの構造（将来実装）

```typescript
interface LicenseToken {
  userId: string;
  plan: 'free_trial' | 'monthly' | 'annual' | 'lifetime';
  issuedAt: number;       // Unix timestamp
  expiresAt: number;      // オフライン猶予の期限
  subscriptionExpiresAt: number;  // サブスクリプション自体の期限
  features: string[];     // 有効な機能フラグ（将来のDLC等）
  signature: string;      // サーバー秘密鍵による署名
}
```

#### データの責務分離

| データ | 保存場所 | 理由 |
|--------|---------|------|
| **セーブデータ（GameState）** | ローカル（IndexedDB / SQLite） | ゲームの応答性。オフラインプレイ必須 |
| **ライセンストークン** | ローカル（暗号化して保存） | オフライン猶予判定に必要 |
| **ユーザーアカウント情報** | サーバー | 認証の正規ソース |
| **課金状態** | サーバー | 課金プラットフォーム連携 |
| **クラウドセーブ（v2）** | サーバー（S3等） | バックアップ・デバイス間同期 |
| **プレイ統計（v2）** | サーバー | ランキング・分析 |
| **マスターデータ** | クライアントにバンドル | オフラインで動作させるため |

### 12.7 Web版先行 → アプリ化ロードマップ

#### フェーズ別リリース計画

```
v1.0  Web MVP（ブラウザ完結）
│     - ゲームエンジン完成
│     - ローカルセーブ
│     - 認証/課金なし（フリープレイ or 限定公開）
│
v1.5  Web + 認証/課金
│     - バックエンド構築（認証API + ライセンスAPI）
│     - LicenseManager + LicenseGate 実装
│     - Stripe連携（Web課金）
│     - PWA対応（Service Worker + オフラインキャッシュ）
│
v2.0  アプリ化 + クラウドセーブ
│     - Capacitorでラップ → iOS/Android
│     - StorageAdapter: SQLite実装
│     - Apple IAP / Google Play Billing 連携
│     - クラウドセーブ同期
│     - プッシュ通知（イベントリマインダー等）
│
v2.5  デバイス間同期 + 追加コンテンツ
│     - マルチデバイス対応（同一アカウントでデータ同期）
│     - DLC / 追加シナリオ（ライセンストークンのfeatures制御）
│
v3.0  オンライン対戦
      - リアルタイム対戦基盤
      - ランキングシステム
```

#### Capacitorアプリ化の前提条件

| 条件 | 対応方針 |
|------|---------|
| **エンジンのDOM非依存** | Phase 1 で徹底。`src/engine/` はReact/DOMをimportしない |
| **Storage抽象化** | StorageAdapterインターフェースでIndexedDB/SQLiteを切り替え |
| **画面サイズ対応** | レスポンシブUI + タッチ操作対応 |
| **ネイティブ機能** | Capacitorプラグインで対応（プッシュ通知、生体認証等） |
| **アプリストア審査** | 課金はIAP経由必須（Apple/Google規約）。Web版とは別課金フロー |
