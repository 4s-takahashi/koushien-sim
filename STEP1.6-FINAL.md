## 1ページサマリー

### 何を作るか
**「同じ打球が二度と起きない、過程にドラマがある、能力差が手触りに出る」高校野球体験**を、軽量で説得力のある物理試合基盤として実装する。

### なぜ作るか（現状の問題）
- v0.42.0 の `out/safe = throwEnd ± 150ms` ハック → 結果ファースト・物理は後付け
- runner.ts に防衛コードが多数（不変条件が保証されていない）
- UI 側の build*Sequence が独自に物理再計算している
- 結果テーブル抽選型に近く、「同じ打席の繰り返し」感が出る

### 設計の核（10項目）
1. **性能値は設計目標**（保証値ではない、Phase R1/R2 後に benchmark 再計測）
2. **投球の3D軌道は持たない**が、**打者認知抽象品質**（perceivedPitchQuality）は持つ
3. **Play Resolver を 6 サブモジュール**に分割（神クラス化を構造的に防止）
4. **走者判断に decisionMargin**（能力・采配・状況で揺れる判断、独立 AI 化可能）
5. **25 入力 → 中間潜在量 5 軸 → 4 軸打球パラメータ**の二段構造（チューニング容易）
6. **21 種分類の品質条件を 4 段階に分割**（存在 / 頻度 / 安定 / 希少）
7. **UI 側は「結果決定ロジック禁止」**（補間・easing・カメラ追従は許可）
8. **runner / inning / result の責務対応表**を明示（防衛コード削除）
9. **Timeline の Replay / Resume / Debug 価値**を設計に組み込み
10. **テスト 5 層化**（物理 / 整合 / 分布 / 多様性 / 再生 E2E）

### 6 レイヤー構成
```
L6 Match Orchestrator   ← 既存 runner.ts のインタフェース維持
L5 Play Resolver         ← 6 サブモジュール分割
L4 Ball Trajectory       ← 解析式 O(1)
L3 Bat-Ball Physics      ← 25入力→潜在量5軸→4軸打球パラメータ
L2 Player Movement       ← 直線等加速度モデル
L1 Field Geometry        ← 静的座標系
```

### Phase 計画（合計 7-8 週間、フェーズ単位リリース可）
- **R1**: 軽量物理基盤（1週、テスト30件以上） — 物理ユーティリティ単体
- **R2**: Bat-Ball Physics（半週、テスト40件） — 中間潜在量二段構造
- **R3**: Play Resolver（1.5週、テスト80件） — 6 サブモジュール
- **R4**: 既存 engine 統合（1週） — 851 件パス維持、防衛コード削除
- **R5**: UI 再生統一（1週） — v0.42.0 ハック削除、Viewer 整合テスト
- **R6**: 表現拡張（1週） — 21 種実況・演出
- **R7**: 戦術・感情接続（1週） — 既存心理システム + NarrativeHook
- **R8**: バランス調整（独立、1-2週） — 1000 試合シミュ + 統計ダッシュボード

→ **Phase R4 完了時点で本番投入可能**（互換性を壊さず置換完了）。
